## 設計決策與 rationale

- **會計模型＝權責認列，但「分錄推算」、不建傳票表**（與 CPA 確認）。理由：完整雙式傳票總帳工程量大；現有 finance+receivables 已能用報表推算借貸與攤提。詳見 user memory [[club-financial-accrual-redesign]]。
- **「每月1日自動轉沖預收→收入」不需排程**。關鍵發現：[PrepaidIncome.vue](../../client/src/pages/PrepaidIncome.vue) 與 [Summary.vue](../../client/src/pages/Summary.vue) 的攤提是**即時推算**（依 startPeriod~endPeriod 平均分攤到所選月份），月度轉沖在報表上本來就是自動的。只有要「實際過帳傳票」才需 cron。
- **批次產生只處理收入型帳款（sourceType='dues'）**：社費、紅箱等都走這條，沖帳會產生 finance income 並依季別攤提。代收（sourceType='agency'）維持原 agency_collections 專屬流程（有結算語意，不混入）。
- **部分收款用累計 paidAmount + status='partial'**：未收 = amount − paidAmount。collect 端點以 `Math.min(輸入, 未收)` 沖抵，避免單筆溢收。
- **階段4（報表改權責認列＋遷移）刻意拆成 4a/4b**：4a（收入頁降級）安全可逆，已做；4b（報表認列來源從 finance 改 receivables ＋既有資料去重遷移）會改寫 CPA 實帳數字、且動正式資料、我看不到資料內容 → 約定先 commit/測試，4b 之後用唯讀查詢盤點再做。**尚未執行 4b。**
- **Zeabur 環境變數用 `${MARIADB_*}` 引用**而非寫死：密碼永遠跟資料庫一致，無同步時間差，也不再有任何寫死帳密。

## 踩過的坑與解法

| 症狀 | 原因 | 解法 / 避雷 |
| --- | --- | --- |
| 移除 db.js fallback 後，部署恐起不到 | 原本 app 其實是**靠寫死 fallback** 連 DB（使用者變數只有 JWT_SECRET/PASSWORD/PORT，無 DB_*） | Zeabur 已自動注入 `MARIADB_HOST/PORT/USERNAME/PASSWORD/DATABASE`；把 DB_* 設成 `${MARIADB_*}` 引用即可 |
| 使用者建了 `PASSWORD` 變數但程式讀不到 | 程式讀的是 `DB_PASSWORD`，名稱不符 | 一律用程式期望的確切名 DB_HOST/DB_PORT/DB_USER/DB_PASSWORD/DB_NAME |
| 「改 Zeabur DB 服務的密碼變數」對既有 DB 無效 | MySQL/MariaDB 的 root 密碼 env 只在**資料目錄首次初始化**時生效 | 真要換密碼得對 DB 下 `ALTER USER ... IDENTIFIED BY ...; FLUSH PRIVILEGES;` 後再同步變數 |
| Express 路由遮蔽風險 | 新增 `/receivables/:id`、`/:id/collect`、`/batch-generate` 等 | 確認段數不同不互相遮蔽：`/:id`(1 段) ≠ `/:id/waive`(2 段)；specific path 先註冊 |
| Bash 工作目錄殘留在 client/ | 早前 `cd client` 後路徑持續 | 後續一律用絕對路徑 `/c/Code/club-financial/...` |

## 給接手伙伴的最重要 invariant

1. **環境變數缺一即拒啟動**（db.js / index.js 開頭）——這是刻意的。本機要有完整 `.env`，Zeabur 用 `${MARIADB_*}` 引用。別再加寫死 fallback。
2. **收入型帳款 sourceType='dues'，代收 sourceType='agency'**。collect 對 agency 只更新 paidMembers 不產生收入；對 dues 產生 finance income 並依類別名「X-Y月」攤提（`getAutoPeriodServer`）。
3. **部分收款**：未收 = amount − paidAmount；status pending→partial→paid 由 `newPaid >= total` 決定。已收款(paidAmount>0)的帳款不可編輯/刪除，需先 reopen。
4. **目前仍是「收款時認列收入」**（modified cash），不是完整權責。Summary/PrepaidIncome 靠 finance 的 startPeriod~endPeriod 即時攤提。改權責＝階段 4b，動報表與遷移前務必先盤點正式資料。
5. **schema 變更走 db.js 的 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`**（MariaDB 支援），純擴充、不破壞既有資料；不要用會掉資料的 migration。
